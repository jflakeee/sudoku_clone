### 3.2 기능 2: 다양한 숫자판 크기 모드

#### 개요
4x4, 6x6, 12x12, 16x16 등 다양한 보드 크기를 지원하는 모드.

#### 기술적 난이도: **상 (High)**

#### 핵심 문제: 9x9 하드코딩 제거

현재 코드는 `9`라는 숫자가 10개 이상의 파일에 하드코딩되어 있음.
보드 크기를 파라미터화하려면 **핵심 엔진 전체를 리팩터링**해야 함.

#### 크기별 블록 구조

| 보드 크기 | 블록 크기 | 숫자 범위 | 총 셀 수 |
|-----------|----------|----------|---------|
| 4x4 | 2x2 | 1-4 | 16 |
| 6x6 | 2x3 | 1-6 | 36 |
| 9x9 | 3x3 | 1-9 | 81 |
| 12x12 | 3x4 | 1-12 (1-9+A-C) | 144 |
| 16x16 | 4x4 | 1-16 (1-9+A-G) | 256 |

#### 변경 범위

| 파일 | 변경 내용 | 크기 |
|------|----------|------|
| `core/solver.js` | `isValid()`, `getCandidates()`, `solve()` → size/blockSize 파라미터화 | 대 |
| `core/generator.js` | `generatePuzzle()` → size 파라미터, DIFFICULTY_RANGES 크기별 조정 | 대 |
| `core/validator.js` | `getRowCells()`, `getColCells()`, `getBlockCells()` → 파라미터화 | 대 |
| `core/scorer.js` | 크기별 점수 배율 추가 | 소 |
| `game/board.js` | Board 생성자에 size 파라미터, `_emptyGrid()` 동적화 | 대 |
| `game/notes.js` | 메모 데이터 구조 크기 대응 | 중 |
| `game/hints.js` | 힌트 로직 크기 대응 | 중 |
| `ui/grid.js` | DOM 동적 생성 (NxN), CSS grid 동적 설정 | 대 |
| `ui/numberpad.js` | 버튼 수 동적 (4, 6, 9, 12, 16개) | 중 |
| `ui/highlight.js` | 하이라이트 범위 동적화 | 중 |
| `css/grid.css` | grid-template 동적화, 반응형 크기 재계산 | 대 |
| `app.js` | 키보드 입력 범위 동적화 (Arrow 범위, 숫자 키 범위) | 중 |

#### 새로 필요한 파일
- `src/js/core/board-config.js` — 보드 크기별 설정 (size, blockRows, blockCols, numRange, difficultyRanges)

#### 선행 작업
- 기능 1 (모드 선택 UI)
- 핵심 엔진 리팩터링 (9 → configurable size)

#### 위험 요소
- **리팩터링 범위가 매우 넓음**: 10개 이상 파일에 하드코딩된 9를 모두 파라미터화
- **12x12, 16x16 퍼즐 생성 성능**: backtracking solver가 큰 보드에서 급격히 느려질 수 있음
  - 16x16: 256셀, 해 탐색 시간 수 초~수십 초 예상
  - Web Worker 사용 또는 사전 생성 전략 필요
- **UI 반응형 문제**: 16x16 보드를 모바일 화면에 표시하기 어려움
- **기존 테스트 전면 수정**: 9x9 가정 테스트 전체 리팩터링 필요
- **12/16x16 숫자 표현**: 10 이상은 A-G 또는 10-16 표기 필요 → numberpad UI 대폭 변경

#### 평가
> **기존 코드 대비 가장 큰 변경**. 핵심 엔진 전체 리팩터링 필요. 단계적 접근 권장:
> 1단계: 4x4 지원 (검증 목적, 블록=2x2)
> 2단계: 6x6 지원 (블록=2x3, 비정사각 블록)
> 3단계: 16x16 (성능 최적화 필수)

## 5. 기술 스택 및 구현 사례 조사

### 5.1 다양한 보드 크기 — 참고 프로젝트 및 알고리즘

| 프로젝트 | 기술 | 지원 크기 | 특징 |
|---------|------|----------|------|
| [Interactive-Sudoku-Solver](https://github.com/sigh/Interactive-Sudoku-Solver) | JavaScript | ~16x16 | 다양한 변형(Killer, Thermo 등) 지원 |
| [hexadoku](https://github.com/fedorkanin/hexadoku) | Python, DLX | 16x16 | Dancing Links(DLX) 기반, 대형 보드 최적화 |
| [@algorithm.ts/sudoku](https://www.npmjs.com/package/@algorithm.ts/sudoku) | TypeScript | (a*a)×(a*a) | 유일해 보장, 범용 크기 지원 |
| [sudoku-puzzle (npm)](https://github.com/DhanushNehru/sudoku-puzzle) | JavaScript | 9x9, 16x16 | `generateSudoku(size, complexity)` API |

**알고리즘 접근법:**
- **백트래킹**: 9x9에서 효율적이나, 16x16 이상에서는 DLX(Dancing Links) 필수
- **DLX (Algorithm X)**: 스도쿠를 Exact Cover 문제로 변환. 16x16+ 보드에서 백트래킹 대비 수십 배 빠름
- **6x6 비정사각형 블록**: `blockRows=2, blockCols=3` 파라미터 분리 필요

### 5.2 타이머 — 정밀 구현 기법

**현재 프로젝트**: `setInterval(1000ms)` + `performance.now()` 기반 (이미 드리프트 보정됨)

**권장 개선**: 카운트다운 시 10초 미만 구간에서 `requestAnimationFrame` 전환
```
setInterval(1초) → 일반 구간 표시
requestAnimationFrame → 10초 미만 긴장감 구간 (100ms 단위 업데이트)
```

**참고**: [game-timer](https://github.com/jensarps/game-timer) — 렌더 루프 연동 타이머

### 5.3 모드 선택 UI — UX 패턴

**권장 구현**: CSS Grid 카드 레이아웃 (현재 다크모드 CSS 변수 시스템 활용)
- 주요 모드(클래식 9x9) → 대형 카드 (`grid-column: span 2`)
- 보조 모드(타임어택, 미니 등) → 일반 카드
- 3~5개 모드가 적정 범위

**참고**: [Game UI Database - Mode Select](https://www.gameuidatabase.com/index.php?scrn=2), Chess.com 모드 선택 UI

## 8. 결론

### 제안된 3가지 기능
- **기능 1 (클래식 모드)**과 **기능 3 (타임어택)**은 현재 아키텍처에서 자연스럽게 확장 가능하며 즉시 구현을 권장합니다.
- **기능 2 (다양한 크기)**는 55곳+ 하드코딩 리팩터링이 필요하며, 16x16은 DLX 알고리즘 도입이 필수입니다. 단계적으로 접근 권장.

### 추가 제안 (서버 없이 즉시 구현 가능)
- **대각선/하이퍼 스도쿠**: 최소 노력 최대 효과. validator 제약 추가만으로 구현.
- **포그 오브 워 스도쿠**: 디지털 전용 차별화 기능. 최우선 추천 변형.
- **스트릭 강화 + 테마/스킨**: 게임화 요소로 리텐션 향상.
