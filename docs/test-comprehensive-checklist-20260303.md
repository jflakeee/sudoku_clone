## 1. 테스트 전략 개요

### 1.1 테스트 피라미드

```
        ┌─────────────┐
        │   수동 테스트   │  크로스 브라우저 / 접근성 / 성능
        ├─────────────┤
        │  E2E 테스트   │  Playwright (메인 레이어) ← 현재 전략
        ├─────────────┤
        │ 통합 테스트    │  기능 간 연동 흐름 (E2E 내 포함)
        ├─────────────┤
        │ 단위 테스트    │  코어 엔진 (validator/solver/generator)
        └─────────────┘
```

- **E2E 중심 전략**: 브라우저 모듈 직접 로드 방식이므로 E2E가 가장 효과적
- **단위 테스트는 E2E 내부에서**: `page.evaluate()` 패턴으로 코어 함수 직접 호출
- Playwright chromium 단일 프로젝트, workers: 1 (순차 실행)

## 2. 회귀 테스트 계획

### 2.1 기존 기능 영향도 매트릭스

신규 기능 추가 시 기존 기능에 미치는 영향을 사전에 식별한다.

| 신규 기능 | 영향받는 기존 기능 | 위험도 | 회귀 테스트 범위 |
|----------|----------------|:-----:|---------------|
| 스트릭 시스템 | 게임 완료(04), 일일도전(06/16), 통계(06) | 중 | R-3, R-5, R-9 |
| XP/레벨 시스템 | 게임 완료(04), 점수(13), 통계(06) | 중 | R-3, R-9, R-2 |
| 보상 시스템 | 업적(09), 완료 화면(04), 설정(06) | 중 | R-3, R-8 |
| 통계 확장 | 통계 화면(06/09), 저장소(05) | 높 | R-9, R-7 |
| UI 개선 | 모든 화면, CSS | 높 | R-1~R-9 전체 |
| 접근성 | 키보드(08), 네비게이션(01), 모든 화면 | 중 | R-1, R-2, R-8 |
| storage 마이그레이션 v2→v3 | 모든 저장 기능(05/14) | 높 | R-7, I-3 |

### 2.2 Phase별 회귀 테스트 범위

#### Phase 1 완료 시 (스트릭 + XP/레벨)
- **필수 통과**: 01~08 (기본 게임플레이 전체)
- **중점 검증**: 04 (게임완료), 05 (저장/복원), 06 (화면 콘텐츠)
- **추가 검증**: 14 (히스토리), 19 (대각선 변형)

#### Phase 2 완료 시 (보상 + 통계 확장)
- **필수 통과**: 01~19 전체 148개
- **중점 검증**: 09 (enhanced UI/어워드), 14 (히스토리), 17 (인쇄)
- **추가 검증**: storage 마이그레이션 테스트

#### Phase 3 완료 시 (UI/접근성/성능)
- **필수 통과**: 01~19 전체 + 신규 전체
- **중점 검증**: 07 (다크모드), 08 (키보드), 09 (UI)
- **추가 검증**: 크로스 브라우저, 반응형

### 2.3 크리티컬 패스 테스트 (반드시 통과)

모든 Phase에서 아래 테스트가 통과해야 배포 가능:

- [ ] CP-1: 새 게임 생성 → 플레이 → 완료 (클래식/타임어택)
- [ ] CP-2: 게임 저장 → 복원 → 계속하기
- [ ] CP-3: 일일 도전 → 완료 → 스트릭 업데이트
- [ ] CP-4: 히스토리 저장 → 재도전 → 완료
- [ ] CP-5: 대각선 변형 → 생성 → 완료
- [ ] CP-6: storage 마이그레이션 후 기존 데이터 보존
- [ ] CP-7: 인쇄 화면 → 레이아웃 → 렌더링
